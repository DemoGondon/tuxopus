# TUXOPUS

TUXOPUS — программный комплекс для автоматизированного обновления программного обеспечения на Linux-серверах с контролем работоспособности, поддержкой отката изменений и формированием отчётов.

Комплекс предназначен для централизованного управления обновлениями в серверной инфраструктуре и позволяет выполнять обновление пакетов на нескольких узлах одновременно с последующей проверкой состояния сервисов и приложений.

---

## Основные возможности

* Обновление одного или нескольких пакетов.
* Массовое обновление на группе серверов.
* Автоматическое определение менеджера пакетов.
* Поддержка:

  * Pacman
  * APT
  * DNF
  * YUM
  * Zypper
* Обновление пакетов безопасности отдельным режимом.
* Перезапуск сервисов после обновления.
* Проверка состояния служб через systemd.
* HTTP и TCP health-check.
* Автоматический откат при обнаружении ошибки.
* Использование системного кэша пакетов для восстановления.
* Дополнительный архивный кэш TUXOPUS.
* Формирование JSON-отчётов по каждому пакету и узлу.

## Архитектура

```
                +----------------+
                | Control Node   |
                |   TUXOPUS      |
                +--------+-------+
                         |
                         | SSH
                         |
        +----------------+----------------+
        |                |                |
        v                v                v

   +---------+     +---------+     +---------+
   |  WEB    |     |  APP    |     |   DB    |
   | Server  |     | Server  |     | Server  |
   +---------+     +---------+     +---------+

```

Управляющий узел запускает Ansible Playbook, который выполняет обновление на целевых серверах без установки дополнительных агентов.

---

## Алгоритм работы

1. Пользователь запускает команду TUXOPUS.
2. Выполняется подключение к выбранным серверам.
3. Определяется используемый менеджер пакетов.
4. Сохраняются версии пакетов до обновления.
5. При необходимости обновляется кэш репозиториев.
6. Выполняется установка обновлений.
7. Новые пакеты сохраняются в архивный кэш TUXOPUS.
8. Перезапускаются указанные сервисы.
9. Выполняются проверки:

   * статус systemd-сервисов;
   * HTTP health-check;
   * TCP health-check.
10. Если проверка не пройдена:

    * запускается процедура отката;
    * восстанавливаются предыдущие версии пакетов.
11. Формируются JSON-отчёты.
12. Пользователь получает результат операции.

## Структура отчётов

После каждого запуска создаётся каталог:

```
reports/
└── 20260524_090103/
    ├── web_nginx.json
    ├── web_curl.json
    ├── app_nfs-utils.json
    └── db_postgresql.json
```

Пример содержимого отчёта:

```
{
  "node": "web-server",
  "package": "nginx",
  "version_before": "1.28.0-1",
  "version_after": "1.28.0-2",
  "verify_ok": true,
  "rollback_triggered": false
}
```

## Примеры использования

### Обновление одного пакета

```bash
tuxopus -n web -p nginx
```

### Обновление нескольких пакетов

```bash
tuxopus -n web -p "nginx,curl,openssl"
```

### Обновление на нескольких серверах

```bash
tuxopus -n "web,app,db" -p curl
```

### Перезапуск сервиса после обновления

```bash
tuxopus -n web -p nginx -s nginx
```

### HTTP health-check

```bash
tuxopus -n web \
        -p nginx \
        -s nginx \
        -c http:80
```

### TCP health-check

```bash
tuxopus -n db \
        -p postgresql \
        -s postgresql \
        -c tcp:5432
```

### Обновление безопасности

```bash
tuxopus -n web --security
```

### Принудительный выбор менеджера пакетов

```bash
tuxopus -n db \
        -p postgresql \
        -m dnf
```

## Параметры командной строки

| Параметр   | Описание                  |
| ---------- | ------------------------- |
| -n         | список узлов              |
| -p         | список пакетов            |
| -s         | список сервисов           |
| -r         | обновить кэш репозиториев |
| -c         | health-check              |
| -b         | размер пакета узлов       |
| -m         | менеджер пакетов          |
| --security | обновление безопасности   |
| -h         | справка                   |
